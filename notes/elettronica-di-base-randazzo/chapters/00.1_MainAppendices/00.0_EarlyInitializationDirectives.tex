% ==============================================================================
% BUGFIX: OVERFULL HBOX IN BIBLIOGRAFIA E CONFLITTO PACCHETTO ULEM
% ==============================================================================
% COMANDO: \PassOptionsToPackage{normalem}{ulem}
% POSIZIONE CORRETTA: Pre-Preambolo (Rigorosamente PRIMA di \documentclass)
%
% 1. IL PROBLEMA (OVERFULL \HBOX IN BIBLIOGRAFIA)
% Durante la compilazione, i titoli lunghi dei libri nella bibliografia
% causavano gravi warning di "Overfull \hbox", sbordando oltre il margine
% destro della pagina. Inoltre, i titoli apparivano sottolineati anziché
% nel classico corsivo tipografico standard.
%
% 2. LA CAUSA (SOVRASCROTTIRA DI \EMPH)
% La custom class del progetto (HKNdocument.cls) carica internamente il
% pacchetto 'ulem' tramite il comando \RequirePackage{ulem}. Di default,
% questo pacchetto sovrascrive i comandi standard \emph e \em, sostituendo
% il corsivo con la sottolineatura continua, precisamente sostituendoli con
% lo stesso comando \uline propsto dal pacchetto ulem stesso. Questo vincolo
% rigido impedisce al motore di sillabazione di LaTeX di spezzare la
% frase a fine riga quado viene usato \emph, in quanto è generato come un box
% insecabile, generando l'errore di overfull.
%
% 3. LA SOLUZIONE E L'EFFETTO DI 'NORMALEM'
% L'opzione 'normalem' (Normal Emphasis) istruisce il pacchetto 'ulem' a
% non sovrascrivere il comportamento di \emph, ripristinando i corsivi
% originali e, di conseguenza, la corretta sillabazione a fine riga.
% Questo non causa alcuna perdita di features, visto che \uline è sempre
% provvisto dal pacchetto.
%
% 4. NESSUN EFFETTO COLLATERALE NELLA RISOLUZIONE DEL BUG
% Analizzando la classe HKNdocument.cls, si evince che il pacchetto ulem
% è stato caricato dal project manager al solo scopo di definire la macro
% \myuline. I corsivi della classe sono stati realizzati in modo sicuro tramite
% \textit. Questa iniezione è quindi priva di effetti collaterali: corregge il
% bug della bibliografia senza alterare in alcun modo l'estetica o le macro
% progettate per l'associazione.
%
% 5. PERCHE' NEL PRE-PREAMBOLO? (LA GESTIONE DELLA MEMORIA)
% Questo comando deve stare prima del \documentclass. Se posizionato nel
% preambolo standard, LaTeX lo ignorerebbe o andrebbe in crash ("Option
% clash"), perché la classe avrebbe già caricato il paccheto ulem in memoria.
% Posizionandolo nel Pre-Preambolo, si lascia un "post-it" al compilatore:
% quando la classe chiamerà ulem, LaTeX gli inietterà di nascosto l'opzione
% 'normalem', aggirando i blocchi di sistema.
%
% 6. ALTRA SOLUZIONE NON PRATICA: MODIFICARE LA CUSTOM CLASS
% Ovviamente per risovlere basterebbe modificare la custom class HKNdocument.cls
% aggiunenddo l'opzione 'normalem' al comando \RequirePackage{ulem}. Tuttavia
% questa soluzione invaliderebbe l'uncità dei file della classe pensata per il
% progetto. Tuttavia si tratta di un bug meritevole di correzione in generale,
% attraverso una pull-request.
%
% 7. ALTRE SOLUZIONI INTERESSANTI:
% - Si avrebbe potuto risolvere anche sfruttando il comando \normalem del
%   pacchetto ulem, che ripstina il comando \emph al suo comportamento normale,
%   in due modi diversi:
%   1.  Mettendolo prima della bibliografia:
%       \normalem  % Comando: Ripristina \emph al suo stato originale (corsivo)
%       \printbibliography
%       \ULforem   % Comando: Riattiva la sottolineatura per \emph (se serve)ù
%   2.  Sfruttando un hook, ossia un'iniezione di codice:
%       \AtBeginBibliography{\normalem}     % lancia \normalem prima di stampare
%                                           % la bibliografia
% - In modo simile, si sarebbe potuto sfruttare un backup del comando \emph
%   originale, fatto con \let\originalemph\emph nel pre-preambolo, per poi
%   risovrascriverlo prima della bibliografia, con \let\emph\OriginalEmph.
% ==============================================================================

% WORKAROUND SFRUTTATO - MECCANISO DI RISOLUZIONE::
% 1. La classe 'HKNdocument' implementa logiche condizionali (es. \ifx)
%    basate sulla variabile \DocumentLanguage. Passare una lista concatenata di
%    lingue direttamente alla classe corromperebbe l'esecuzione di tali script

% 2. Demandando l'opzione tramite \PassOptionsToPackage, il compilatore LaTeX
%    memorizza l'argomento. Al momento dell'effettiva invocazione di 'babel'

%    merge delle opzioni.
% 3. L'elenco finale risolto dal compilatore risulta essere [english, italian].

%    in memoria come Secondary Languages.
%
% OSSERVAZIONE - SINERGIA TRA BABEL E MOTORE BIBLIOGRAFICO:
%                       LOCALIZZAZIONE VS SILLABAZIONE
% Questa pre-configurazione è un prerequisito tecnico necessario per il corretto
% funzionamento generale del pacchetto 'biblatex', se inizializzato con
% l'ozpione "autolang=other' o "autolang=hyphen".
% In precisione hyphen chiama solo la sillabazione, mentre autolang chiama sia
% la sillabazione che la localizzazione (spiegazione a seguire), nella lingua
% specificata in "langid={...}" all'interno del database bibliografico (.bib).
%
% I due processi elaborativi separati da considerare sono:
% - LOCALIZZAZIONE: La traduzione delle etichette bibliografiche (es. "6th Ed."
%   invece di "6ª Ed.") è gestita autonomamente dai dizionari interni di
%   'biblatex' tramite la lettura del metadato 'langid={english}'.
% - SILLABAZIONE (Hyphenation): 'biblatex' non possiede algoritmi di andata
%   a capo, ma demanda interamente questa operazione ai pattern caricati in
%   memoria dal pacchetto 'babel'.
%
% Omettere l'istruzione \PassOptionsToPackage genera un'anomalia tipografica

% inglese, ma il compilatore applicherà le regole di sillabazione italiane ai
% testi anglosassoni, causando potenziali cesure errate (non grammaticali)
% nell'andata a capo a fine riga.

% ------------------------------------------------------------------------------
% 0. IL LATEX TAGGING PROJECT (Cos'è e perché nasce)
% ------------------------------------------------------------------------------
% Questo comando è la porta d'accesso al "LaTeX Tagging Project", un colossale
% sforzo pluriennale del team LaTeX3 (introdotto in fase matura a partire
% dalle distribuzioni TeX Live 2023). L'obiettivo è colmare il divario storico
% tra la tipografia classica di TeX e i moderni standard digitali
% internazionali, rendendo i PDF nativamente accessibili (PDF/A, PDF/UA),
% leggibili dagli screen reader per non vedenti ed esportabili fluidamente in
% altri formati (es. HTML/XML).
%
% ------------------------------------------------------------------------------
% 1. COSA AGGIUNGE E COSA SOVRASCRIVE (SOTTO IL COFANO)
% ------------------------------------------------------------------------------
% - Gestione Sillabazione (Soft Hyphen vs Hard Hyphen):
%   Di default, il vecchio LaTeX sillaba le parole inserendo un "Hard Hyphen"
%   (Trattino rigido, U+002D in codifica Unicode/UTF-8), che spezza la parola
%   in modo permanente. Questo nuovo comando lo sostituisce con il "Soft
%   Hyphen" (Trattino opzionale, U+00AD). Questo carattere invisibile dice al
%   lettore PDF: "Mostra graficamente un trattino qui, ma la parola resta
%   logicamente intera". Ciò consente ad esempio di poter trovare, con lo
%   strumento di ricerca interno ai lettori pdf, perlomeno a quelli col
%   supporto al soft-hyphen (vedi sezione più avanti sulla compatibilità), le
%   parole anche se stampate in versione sillabata.
% - Metadati Centralizzati: Sostituisce il "Far West" in cui ogni pacchetto
%   (come hyperref o pdfx) scriveva a modo suo le info nel PDF. Ora c'è un
%   dizionario centrale standardizzato per titolo, autore, lingua e standard
%   ISO.
% - PDF Strutturato (Tagged PDF): Aggiunge un livello logico (Tag) nascosto
%   dietro la grafica, indicando ai software dove sono i titoli, i paragrafi
%   e le tabelle.
% - Gestione Colore Avanzata: Inizializza un nuovo backend per la fedeltà
%   cromatica.
%
% ------------------------------------------------------------------------------
% 2. EFFETTI COLLATERALI E ATTENZIONI DA AVERE
% ------------------------------------------------------------------------------
% - [!] Il Problema dei Font: Alcuni font custom o scaricati da internet non
%   sono programmati bene e mappano il carattere U+00AD come "invisibile" o a
%   larghezza zero. Se usando questo comando si nota che i trattini a fine
%   riga spariscono visivamente, la colpa è del font. Usando font di base
%   robusti (come il Latin Modern standard di LaTeX) il problema non sussiste
%   mai.
% - Peso e Compilazione: L'aggiunta di strutture logiche e dizionari rende il
%   PDF leggermente più pesante in termini di Kilobyte/Megabyte, e i tempi di
%   compilazione aumentano di qualche frazione di secondo.
% - Rischio Conflitti: Poiché sovrascrive il motore base, vecchi pacchetti
%   non aggiornati (che usano primitive obsolete come \pdfobj) causeranno
%   errori fatali.
%
% ------------------------------------------------------------------------------
% 3. COMPATIBILITÀ DEI LETTORI PDF (Focus sui Soft Hyphen e Ricerca Testo)
% ------------------------------------------------------------------------------
% Il PDF generato è strutturalmente perfetto, ma la resa finale (stampa del
% trattino, ricerca della parola intera, copia-incolla pulito e aspetto dei
% segnalibri) dipende interamente dall'intelligenza del Reader utilizzato
% dall'utente finale:
%
% * Adobe Acrobat / Foxit / PDF-XChange / Okular (I Gold Standard)
%   - Ricerca: PERFETTA. Trovano la parola intera ignorando l'andata a capo.
%   - Copia-Incolla: PERFETTO. Ricuciono la parola senza incollare trattini.
%   - Segnalibri: PERFETTI. Il trattino invisibile non viene mostrato
%     nell'indice.
%   - (Nota: Okular, basato sul motore Poppler, è l'alternativa open-source
%     più vicina alla perfezione, leggerissimo sia su Linux che su Windows).
%
% * Google Chrome / Edge / Firefox (Web Viewers)
%   - Ricerca: OTTIMA. Trovano quasi sempre la parola per intero.
%   - Copia-Incolla: BUONO. Comportamento simile a VS Code o perfetto a
%     seconda del sistema operativo sottostante.
%   - Segnalibri: OTTIMI. Nessun difetto visivo nell'indice laterale.
%
% * Sioyek (L'alternativa leggera per l'Accademia)
%   - Ricerca: OTTIMA. Sfruttando un motore MuPDF moderno e ben integrato,
%     trova correttamente le parole sillabate.
%   - Copia-Incolla: BUONO. Comportamento generalmente corretto
%     nell'estrazione.
%   - Segnalibri: BUONI. Gestisce i titoli nell'indice senza i bug di Sumatra.
%
% * Lettore Interno di VS Code (Basato su PDF.js / Mozilla)
%   - Ricerca: OTTIMA. Il motore normalizza il testo e trova le parole
%     sillabate.
%   - Segnalibri: OTTIMI. Il trattino rimane invisibile nell'indice laterale.
%   - Copia-Incolla: IMPERFETTO. Copiando il testo, il carattere invisibile
%     U+00AD finisce negli appunti. Se incollato altrove, la parola sembra
%     unita ma contiene un carattere "fantasma" in mezzo.
%
% * SumatraPDF (L'anello debole: lettore leggero e veloce, man non supportato)
%   - Ricerca: ROTTA. Non riesce a trovare la parola spezzata tra due righe.
%   - Segnalibri: ROTTI. Se un titolo ha un soft hyphen, Sumatra stampa
%     visivamente il trattino in mezzo alla parola nell'indice laterale.
%   - Copia-Incolla: ROTTO. Spezza la parola e incolla il trattino fisico.
%
