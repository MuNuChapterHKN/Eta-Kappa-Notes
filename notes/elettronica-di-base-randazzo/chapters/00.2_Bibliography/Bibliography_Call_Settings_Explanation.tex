    % BibLaTeX è un pacchetto per la gestione avanzata della bibliografia,
    % standard moderno per la bibliografia in LaTeX.
    % - backend=biber: Specifica il motore bibliografico da utilizzare per
    %   l'elaborazione dei dati contenuti nel database bibliografico (.bib),
    %   in questo caso generato con Zotero.
    %   Nelle distribuzioni moderne di TeX, in precisione TeX Live, è già
    %   impostato di default, tuttavia esplicitarlo è la best practice, e
    %   garantisce la compatibilità anche con distribuzioni più datate.
    %   Offre pieno supporto nativo al formato UTF-8 e algoritmi di ordinamento e
    %   parsing molto più avanzati del legacy BibTeX.
    %
    % - autolang=other: Risolve i problemi di sillabazione della lingua nella
    %   bibliografia, nel caso le fonti citate abbiano più lingue diverse.
    %   È fondamentale che il campo "langid" nel database .bib sia esplicitato
    %   correttamente, in quanto è il campo usato per selezionare come viene
    %   trattata ogni voce della bibliografia.
    %   Zotero trasforma la lingua in langid automaticamente, se è scritta
    %   correttamente (codici ISO, o per esteso in inglese).
    %   In precisione "other" crea un ambiente linguistico isolato per la singola
    %   riga della bibliografia: adatta sia le regole di sillabazione che di
    %   formattazione e di terminologia funzione della localizzazione (esempio:
    %   Edition vs Edizione, a cura di vs edited by, eccetera) garantendo le
    %   migliori impostazioni tipografiche per ogni voce.
    %   Come alternativa si può usare "autolang=hyphen", che si limita solo alla
    %   sillabazione multilingua, senza adattare il resto.
    %
    % - style=ieee: applica lo standard bibliografico dell'Institute of Electrical
    %   and Electronics Engineers (numerico con parentesi quadre, titoli
    %   formattati e abbreviazioni specifiche). In precisione:
    %     * Citazioni testuali: numeriche tra parentesi quadre (es. [1]),
    %     con accorpamento automatico per fonti multiple (es. [1]-[3]).
    %     * Nomenclatura: autori indicati con iniziali del nome e cognome per
    %       esteso (es. J. C. Maxwell).
    %     * Tipografia dei titoli: titoli di articoli o paper racchiusi tra
    %       virgolette, mentre i nomi di riviste, manuali e libri sono formattati
    %       in corsivo.
    %     * Ordinamento bibliografico: rigorosamente sequenziale, basato
    %       sull'esatto ordine di comparsa nel testo (sorting 'none').
    %       Se si forza la stampa delle fonti non citate, vengono elencate dopo
    %       quelle citate, nell'ordine di comparsa nel database .bib.
    %     * Dettagli tecnici: standardizzazione automatica della punteggiatura
    %     e delle abbreviazioni specifiche (Vol., No., pp.).
    %   È la scelta coerente e per l'identità del Mu Nu Chapter HKN.
    %   (Nota: richiede che nel sistema sia installato 'biblatex-ieee', incluso
    %   con TexLive. Se si usa una distribuzione diversa, lo si può installare da
    %   terminale col comando "tlmgr install biblatex-ieee").

\begin{comment}     NOTA IMPORTANTE - BUG DI OVERFULL \HBOX IN BIBLIOGRAFIA CAUSATO DAL PACCHETTO ULEM

    Per le informazioni complete su questo bug, la sua causa e la soluzione adottata, si rimanda al file
    00.0_EarlyInitializationDirectives.tex, dove è stata inserita una spiegazione dettagliata e completa.
    In Sintesi: il pacchetto ulem, utilizzato dalla custom class per creare il comando \myuline (sottolineatura speciale
    interrotta per le parti basse delle lettere (p, g, q, j)) sovrascrive il comando \emph, usato da BibLaTeX per
    formattare i titoli in bibliografia, chiudendo le parole enfatizzate in un box insecabile. Se il titolo è lungo,
    il box non può essere spezzato su più righe, causando l'overfull, con i titoli che sbordano oltre il margine destro.
    Si è risolto semplicemente impedendo a ulem di sovrascrivere \emph tramite l'opzione 'normalem, nel pre-preambolo.

\end{comment}

\begin{comment}     NOTA IMPORTANTE SULLA GESTIONE DELLA BIBLIOGRAFIA

==============================================================================
ARCHITETTURA DI IMPAGINAZIONE E GESTIONE DELLA BIBLIOGRAFIA
==============================================================================
1. GESTIONE DEGLI STATI NELLA CUSTOM CLASS (HKNdocument.cls)
La classe HKNdocument gestisce automaticamente le transizioni di stato
del libro (frontmatter, mainmatter, backmatter) sfruttando un'architettura
a eventi (Event-Driven) tramite hook, ossia punti di iniezione di codice.

- Frontmatter e Mainmatter:
  * Hook usato: \AtBeginDocument
  * Funzionamento: Nell'istante in cui si compila \begin{document}, la
    classe inietta \frontmatter (numeri romani, no capitoli), genera il
    frontespizio, la licenza e l'indice. Subito dopo, innesca \mainmatter
    (numeri arabi e capitoli attivi) per iniziare gli appunti.

- Backmatter:
  * Hook usato: \AfterEndDocument
  * Funzionamento: Quando si compila \end{document}, l'hook sospende lo
    spegnimento di LaTeX, inietta \backmatter e carica il file condiviso
    _postamble.tex.

------------------------------------------------------------------------------
2. LA SCELTA ARCHITETTURALE SULLA BIBLIOGRAFIA
Tipograficamente, la bibliografia appartiene al Backmatter. Poiché la
custom class attiva il \backmatter internamente richiamando il file
_postamble.tex, la soluzione "teorica" perfetta sarebbe inserire
\printbibliography direttamente all'interno di quel file.

Tuttavia, si è preferito NON farlo per questioni di compatibilità e version
control. Il file _postamble.tex è un file centralizzato del Project Manager.
Modificarlo per gestire esigenze locali inquinerebbe la repository e
causerebbe conflitti nei branch degli altri membri del team.

------------------------------------------------------------------------------
3. LA SOLUZIONE LOCALE: INTERRUTTORI IDEMPOTENTI
Per stampare la bibliografia nel main.tex in modo corretto senza toccare i
file di progetto, richiamiamo manualmente il comando \backmatter subito
prima di \printbibliography.

Perché il sistema non va in crash con due chiamate a \backmatter?
I comandi di stato (\frontmatter, \mainmatter, \backmatter) sono
"Interruttori Idempotenti". Eseguirli più volte consecutive produce
esattamente lo stesso risultato della prima esecuzione.

Quando si innescherà l'hook finale del PM (\AfterEndDocument), esso chiamerà
nuovamente \backmatter. A quel punto, LaTeX si limiterà a "spegnere un
interruttore già spento", ignorando la seconda operazione in modo silente
e totalmente sicuro.
